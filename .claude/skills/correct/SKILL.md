# /correct — 訂正ワークフロー

**引数**: slug(公開済み記事のslug。ファイル名は `YYYY-MM-DD-<slug>.md`)、誤りの内容(オーナーからの指摘)

## 手順

1. **該当記事を読み、出典を再調査する**
   - `src/content/articles/YYYY-MM-DD-<slug>.md` を読み、指摘された誤りの該当箇所を特定する。
   - WebSearch / WebFetch で出典を再調査し、正しい情報を確認する。既存の `sources` だけでなく、必要なら追加のソースも確認する。
   - **出典を確認できない訂正は行わない**。確認できなかった場合はその旨をオーナーに報告し、以降の手順は行わない。

2. **本文の該当箇所を修正する**
   - 修正は誤りの箇所に限定する最小限の修正とする。関係ない箇所の書き換え・言い回しの調整は行わない。

3. **訂正履歴を追記する**
   - frontmatterの `corrections` 配列に1件追加する:
     - `date`: 今日の日付
     - `description`: 「いつ・何を・なぜ」が伝わる形式で書く
       (例: 「業況判断DIの改善幅を『6ポイント』と記載していましたが、正しくは『5ポイント』でした。一次ソースの再確認漏れによる転記ミスです。」)
   - `updatedDate` を今日の日付に設定する。

4. **教訓を `editorial/LESSONS.md` に追記する**
   - `/publish` と同じフォーマットを厳守する:
     ```
     - [YYYY-MM-DD][分類: 事実確認] 一般化した教訓1行(具体例: 「X」→「Y」)
     ```
   - 記録するのは**訂正内容そのもの**ではなく、**誤りが発生した原因**(裏取りの手順のどこに落とし穴があったか)。次の `/article` `/publish` で同じ誤りを防げる形に一般化する。

5. **ビルド確認とコミット**
   - `npm run build` がエラー0で通ることを確認する。
   - 記事・LESSONS.mdの変更を `git add` し、コミットメッセージ `correct: <slug>` でコミットする。

6. **pushとオーナーへの報告**
   - **push は他者から見える公開行為であり、実行前に必ずオーナーに最終確認を取ってから行う**。
   - push後、数分待ってから本番URL(`https://<プロジェクト名>.pages.dev/articles/<日付>-<slug>/`)で訂正表示(CorrectionNote、朱の罫線で表示される「訂正」欄)が反映されていることを確認し、オーナーに結果を報告する。

## 注意

- **訂正は隠さない。本文をこっそり直すだけの対応は禁止**(`corrections` 配列への追記が必須。`editorial/POLICY.md` の訂正方針に基づく)。
- LESSONS.mdへの追記は「誤りを咎める」ためではなく「次回の裏取りの精度を上げる」ために行う。原因まで踏まえて一般化すること。

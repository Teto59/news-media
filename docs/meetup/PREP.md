## 2. トーク原稿(約5分・テーブルで話す用)

そのまま読まず、流れだけ頭に入れて自分の言葉で。太字が各ブロックの「言いたいこと1つ」。

### 掴み(30秒)
> 私はコードが一行も書けません。でも、AIに新聞を作らせて運用しています。
> 「Chronica」というニュースサイトです。無料・広告なし・トラッカーなし。記事はClaudeが複数ソースで裏取りして書き、私が添削して公開しています。

### 動機(30秒)
> きっかけは、既存メディアの裏取りの甘さへの不満です。人間の記者は締切に追われて裏取りを省く。でもAIなら「独立した情報源2つ以上・一次ソース優先」を**毎回・機械的に強制できる**のでは、という仮説で始めました。

### 設計の知見①: できないことは、しなくていい設計にする(1分)
> 私はサーバー管理もセキュリティ対策もできません。だから守るものをゼロにしました。データベースなし・サーバーなし・読者の情報を一切持たない静的サイトです。
> 合言葉は**「全データを引っこ抜かれても問題ない」**。漏れて困る情報が最初から存在しないので、構造的に安全です。これはClaudeと相談して決めた、最初で最大の設計判断でした。

### 設計の知見②: 編集ループ(1分半・ここが本体)
> 私のAIへの指示は、チャットではなく**全部日本語のルールファイル**に書いてあります。CLAUDE.mdと編集方針フォルダ。いわばAIにとっての「社則」です。
> 運用はループになっています:
> 1. `/article` — AIが複数ソース裏取り付きの初稿を書く
> 2. 私が添削する
> 3. `/publish` — AIが**私の添削の差分を自分で分析して、教訓をファイルに書き溜める**
> 4. 同じ教訓が3回溜まったら `/retro` で正式ルールに昇格 → 次の初稿が最初から良くなる
>
> つまり私は「直す」だけで、「なぜ直されたか」の言語化と再発防止はAIが自分でやります。運用2日で教訓が12件溜まりました。

### 踏んだ地雷(1分・一番ウケるところ)
> - AIのウェブ要約は、同じ文書に複数の対象が出てくると**取り違える**ことがある。要約を信じず、一次資料そのもので裏取りし直すルールにしました。
> - 「草案と確定版の比較記事」を確定版同士の差だと誤読して、記事のタイトルと骨子を書き直す羽目になりました。出典が「どの時点のテキスト」を比べているかの確認をルール化。
> - 条約や覚書は要約すると「第13項が第1・4・5項を参照する」みたいな構造が壊れる。原番号付き全文掲載が基本、に変えました。

### 締め(30秒)
> 非エンジニアの開発手法は「**自然言語のルールファイル+フィードバックループ**」だと思っています。コードは書けなくても、AIと一緒に品質を積み上げる仕組みは作れる。リポジトリは全部GitHubで公開しているので、見ていってください。

## 3. デモ台本(テーブルで見せる順)

コツ: **デモの操作自体をClaude Codeに日本語で頼む**こと。「非エンジニアがAIを操縦している」姿そのものが一番のデモです。

1. **サイトを見せる(30秒)** — スマホかMacで https://news-media-8c6.pages.dev を開く
2. **「社則」を見せる(1分)** — Claude Codeで「CLAUDE.mdとeditorialフォルダの中身を見せて」と頼む。「私の指示は全部これです」
3. **添削の差分を見せる(2分)** — Claude Codeに「イラン覚書の記事の、初稿コミットと公開版の差分を見せて」と頼む。実際に直した箇所が赤/緑で出る
4. **教訓ファイルを見せる(1分)** — 「LESSONS.mdを見せて」。差分から自動抽出された12件の教訓
5. **(余裕があれば)ライブ実行** — 「/article」でその日のニュースを1本書かせてみる。裏取り検索が走る様子だけ見せれば十分(完成まで待たなくてよい)

## 4. 想定Q&A

- **ハルシネーション(AIのもっともらしい嘘)対策は?**
  → 「独立した情報源2つ以上・一次ソース最優先・確認できない事実主張は書かない」をワークフローに強制。それでも起きた取り違えは教訓化して再発防止(実例: 要約の対象取り違え)。最後の砦は人間の添削。

- **費用は?**
  → ホスティング0円(Cloudflare Pages無料枠)。かかっているのはClaudeのサブスクリプション代だけ。

- **なぜ静的サイト?**
  → 守る情報をゼロにするため。読者データを持たない=漏洩リスクが構造的に存在しない。非エンジニアはセキュリティ運用ができないので、「運用でがんばる」のではなく「構造で解決」した。

- **記事の責任は誰が取る?**
  → 最終添削と公開判断は人間(私)。誤りが見つかったら `/correct` ワークフローで訂正し、訂正履歴もサイトで公開する。

- **本当にエンジニアの助けゼロ?**
  → ゼロ。設計相談もコードも全部Claude Code。私がやるのはニュース選定・添削・公開判断の3つだけ。

- **プロンプトのコツは?**
  → 単発のプロンプトを磨くより、**ルールをファイルに蓄積する**こと。会話は忘れられるが、ファイルは毎回読み込ませられる。

- **使っているモデルは?**
  → Claude Code上で、Sonnet 5で構築を開始し、現在はFable 5も併用。

- **1記事どれくらいかかる?**
  → (自分の体感値をここにメモしておく: 裏取り+執筆で◯分、添削で◯分)
